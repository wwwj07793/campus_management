# 软件能力提升规划

## 个人方向定位

结合电子专业背景和当前学习内容，后续能力建设不应只走纯软件路线，也不应只停留在 LangChain / LangGraph 框架 demo。

更适合的方向是：

```text
硬件数据采集 + Python 后端 + 数据库 + 数据分析 + AI Agent + 可视化平台
```

也就是成为能够把现实世界数据接入 AI 系统的人。

## 当前阶段的主要问题

目前已有基础：

- 学过 LangChain 和 LangGraph
- 校园管理系统项目已完成一部分
- 已经开始理解核心数据、派生数据、测试数据和业务流程

当前迷茫点：

- 校园管理系统、AI Agent、电子硬件之间看起来割裂
- 不清楚软件能力应该练到什么程度
- 容易在框架学习和项目开发之间来回跳

实际上，这三条线可以合并成一条主线：

```text
硬件采集数据
-> 后端管理数据
-> 数据库存储数据
-> 数据分析 / AI Agent 处理数据
-> 生成预警、建议或报告
-> 可视化展示
```

## 必须重点提升的能力

### 1. Python 工程能力

目标不是只会语法，而是能组织一个清晰项目。

需要掌握：

- `models`
- `services`
- `repositories`
- `tests`
- `config`
- `api`

在当前校园项目中重点练：

- 数据模型设计
- 业务逻辑拆分
- 核心数据和派生数据分离
- 自动化测试
- 项目文档

### 2. 数据库能力

硬件 + AI 项目离不开数据存储。

需要掌握：

- 表设计
- 主键、外键
- 一对多、多对多关系
- SQLAlchemy
- Alembic 迁移
- 基础查询和事务

校园项目中的学生、课程、选课、成绩表，未来可以迁移为：

```text
devices
sensor_readings
alerts
control_commands
```

### 3. API 能力

硬件端、前端、AI 服务之间都需要接口连接。

建议学习 FastAPI，并能实现：

```text
POST /students
POST /grades
GET /analytics/gpa
POST /sensor-data
GET /devices
POST /agent/analyze
```

### 4. 数据分析能力

相比继续堆 Agent 框架，当前更应该补数据分析能力。

需要掌握：

- Pandas 数据清洗
- Matplotlib / Plotly 可视化
- 异常检测
- 趋势分析
- 简单预测

校园项目中可以做：

- 成绩分布分析
- GPA 预警
- 学业风险名单

硬件项目中可以做：

- 温湿度异常检测
- 光照趋势分析
- 设备状态预警

### 5. AI Agent 落地能力

LangChain / LangGraph 不应只停留在聊天 demo。

要学会让 Agent 调用真实工具：

- 查询数据库
- 读取传感器数据
- 调用统计函数
- 生成分析报告
- 触发预警逻辑

目标是做出真正连接项目数据的 Agent，而不是孤立的大模型问答。

## 三阶段项目路线

### 第一阶段：稳定校园管理系统后端

目标：练软件基本功。

任务：

1. 学生、课程、成绩核心流程稳定
2. 内存版完整业务测试通过
3. 接入 SQLite 数据库
4. 编写 Repository 层
5. 补 5-10 个 pytest 测试
6. 完善 README

这个阶段先不要追求复杂 AI，重点是把工程底座做好。

### 第二阶段：加入数据分析和 API

目标：让项目从管理系统变成数据平台。

任务：

1. 编写 FastAPI 接口
2. 提供成绩 / GPA 统计接口
3. 提供学业预警接口
4. 使用 Pandas 做分析
5. 使用 Matplotlib / Plotly 输出图表

完成后，项目应具备“数据管理 + 数据分析”的能力。

### 第三阶段：加入硬件模拟和 Agent

目标：对齐电子 + AI 方向。

任务：

1. 增加教室环境模拟数据
2. 设计设备数据表
3. 实现环境数据入库
4. 实现温湿度 / 光照异常检测
5. 使用 LangGraph Agent 查询数据库
6. 让 Agent 生成预警报告

最终项目可以定位为：

```text
校园数据管理与智能预警平台
```

系统支持：

- 学生成绩管理
- 学业风险预警
- 教室环境数据监测
- AI 自动分析报告

- 测试场景怎么设计


## 加入 Agent 后的项目流程框架

LangChain / LangGraph 不应作为孤立 demo 学习，而应该接入当前校园管理项目，形成真实工具调用能力。

加入 Agent 后，项目可以分成四层：

```text
第 1 层：数据层
学生、课程、选课、成绩、教室环境数据

第 2 层：分析层
GPA 统计、成绩分布、学业预警、环境异常检测

第 3 层：工具层
把查询和分析封装成 Agent 可调用工具

第 4 层：Agent 层
用 LangGraph 编排工具调用，并生成中文分析报告
```

核心思想：

```text
Agent 不直接“凭空回答”，而是先调用项目中的真实工具；
工具从真实数据中查询和分析；
Agent 只负责理解问题、选择工具、组织结果、生成报告。
```

### Agent 应该服务的具体任务

不要一开始做万能聊天助手，先做“校园数据分析 Agent”。

它应该能回答：

- 哪些学生有学业风险？
- 某个学生的成绩情况怎么样？
- 哪个院系 GPA 偏低？
- 哪门课不及格率较高？
- 哪个教室环境异常？
- 帮我生成今日校园数据报告。

### 工具函数优先于 Agent 编排

在写 LangGraph 节点前，先写普通 Python 工具函数。

建议新增：

```text
core/agent_tools.py
```

核心工具函数：

```text
get_student_profile(student_id)
get_student_grade_report(student_id)
get_warning_students()
get_department_gpa_summary()
get_course_score_summary(course_id)
get_environment_abnormal_report()
generate_daily_campus_summary()
```

这些函数应该调用：

```text
StudentManage.students_dict
CourseManage.all_courses_dict
StudentManage.data_view()
GradeManage
```

也就是让 Agent 的回答建立在项目真实数据上。

### LangGraph 基础流程

第一版 LangGraph 不需要复杂多智能体，只需要稳定工作流：

```text
用户问题
-> classify_intent：判断问题类型
-> call_tool：调用对应工具函数
-> format_report：整理成中文报告
-> 输出结果
```

可识别的问题类型：

```text
student_report
warning_report
department_summary
course_summary
environment_report
daily_report
unknown
```

### 后续演进路线

Agent 工具的数据来源可以逐步升级：

```text
内存数据工具
-> Repository 数据库工具
-> API 数据工具
-> 真实硬件数据工具
```

这样可以从当前项目自然过渡到真实 AI 应用系统。

### 必须掌握的 Agent 能力

真正重要的不是背框架 API，而是掌握：

- 工具边界设计：一个工具只做一件明确的事
- 输入输出结构化：尽量返回 dict / list，而不是混乱文本
- 状态设计：LangGraph State 中保存问题、意图、工具结果、最终回答
- 错误处理：学号不存在、数据为空、工具失败时要有明确反馈
- 结果校验：Agent 回答必须来自工具结果，不能胡编
- 报告生成：把结构化数据转成清楚的中文分析

### 加入 Agent 后的项目定位

项目可以从：

升级为：

系统能力变成：

## 电子电气相关方向与 AI 的结合点

电子电气背景下，AI 的结合方向不只有芯片。不同方向的门槛、学习重点和落地方式差别较大。

### 三个核心方向

| 方向 | 核心问题 | 典型例子 | 门槛 |
|---|---|---|---|
| AI 理解现实数据并响应 | AI 如何根据现实世界数据做判断、预测、决策 | 成绩预警、设备故障诊断、环境异常检测、工业视觉、机器人感知、车载 AI | 低到高，随场景复杂度上升 |
| AI 与硬件计算体系协同设计 | AI 如何在硬件上跑得快、稳、省电；硬件如何更适合 AI | TinyML、模型量化、边缘 AI、FPGA 加速、NPU、低功耗 AI 芯片 | 中高到很高 |
| AI Agent 协助专业工作流 | AI 如何调用工具，稳定辅助专业任务 | 自动查数据库、生成分析报告、辅助电路设计、通信仿真、芯片验证、电力调度分析 | 取决于专业任务难度 |

一句话概括：

```text
方向 1：AI 看懂现实数据
方向 2：硬件跑好 AI，AI 适应硬件
方向 3：Agent 帮人完成专业任务
```

当前阶段推荐顺序：

```text
先做方向 1
再结合方向 3
最后逐步靠近方向 2
```

### 总览

| 方向 | AI 结合点 | 门槛 | 推荐程度 |
|---|---|---:|---:|
| 电气工程 | 电力预测、故障诊断、智能电网、设备状态监测 | 中 | 高 |
| 通信工程 | 信道估计、信号检测、资源分配、6G 智能通信 | 中高 | 中高 |
| 电子工程 | 传感器数据处理、嵌入式 AI、边缘 AI、智能硬件 | 中 | 很高 |
| 计算机工程（硬件分支） | 计算机体系结构、嵌入式系统、边缘 AI、FPGA/GPU/NPU 加速、硬件软件协同 | 中高 | 高 |
| 芯片方向 | AI 加速器、NPU、EDA AI、存算一体、类脑芯片 | 很高 | 谨慎选择 |

### 电气工程 + AI

主要结合点：

- 电力负荷预测
- 新能源发电预测
- 电网故障诊断
- 变压器、电机状态监测
- 智能配电网
- 储能系统优化
- 电动汽车充电调度
- 工业设备预测性维护

门槛：中等。

需要能力：

- 电路、电机、电力系统基础
- Python
- 数据分析
- 机器学习
- 时间序列预测
- 基础深度学习

适合场景：

如果未来想进入电网、新能源、工业自动化、设备运维等方向，电气 + AI 是比较稳的路线。

### 通信工程 + AI

主要结合点：

- 信道估计
- 信号检测
- 调制识别
- 频谱感知
- 无线资源分配
- 网络流量预测
- 基站能耗优化
- 6G 智能通信
- 雷达通信一体化感知

门槛：中高。

需要能力：

- 概率论
- 线性代数
- 信号与系统
- 数字信号处理
- 通信原理
- Python / MATLAB
- 机器学习
- 深度学习

适合场景：

如果数学基础较好，并且愿意深入信号处理和通信理论，可以考虑通信 + AI。这个方向前景好，但抽象程度较高，研究生阶段更常见。

### 电子工程 + AI

主要结合点：

- 传感器数据处理
- 嵌入式 AI
- 边缘 AI
- 智能检测设备
- 工业视觉
- 机器人感知
- 可穿戴设备
- 环境监测
- 低功耗智能终端

门槛：中等。

需要能力：

- 模电、数电
- 单片机
- 传感器
- C / C++
- Python
- 数据采集
- 基础机器学习
- 边缘部署

适合场景：

可参考项目形态：


### 计算机工程（硬件分支）+ AI

主要结合点：

- 计算机体系结构与 AI 加速
- FPGA / GPU / NPU 上的模型推理加速
- 嵌入式系统上的边缘 AI 部署
- ARM / RISC-V 平台上的模型量化和推理优化
- 存储、缓存、总线对 AI 计算性能的影响分析
- 硬件软件协同设计
- 实时系统中的 AI 推理任务调度
- 边缘计算设备的数据采集、推理和控制闭环

门槛：中高。

需要能力：

- 数字电路
- 计算机组成原理
- 计算机体系结构
- 嵌入式系统
- C / C++
- Python
- Verilog / FPGA 基础
- Linux
- 操作系统基础
- 深度学习基础
- 模型量化、剪枝、部署
- 基本性能分析能力

适合场景：

这里的计算机工程不是软件专业路线，而是偏电子方向的计算机硬件系统路线。它比普通 AI 应用开发门槛更高，但比直接做 AI 芯片架构更现实。

典型项目形态：

```text
摄像头 / 传感器 / 边缘设备
-> ARM / RISC-V / FPGA / Jetson 等硬件平台
-> 模型量化和部署
-> 本地实时推理
-> 控制执行或异常预警
```

适合先从“边缘 AI 部署和硬件加速实验”切入，而不是一开始直接做芯片架构。

### 芯片方向 + AI

- AI 加速器
- NPU
- FPGA 神经网络加速
- EDA + AI
- 存算一体
- 类脑芯片
- 模型量化与硬件部署

门槛：很高。

需要能力：

- 数字电路
- Verilog
- 计算机组成
- 计算机体系结构
- FPGA
- 深度学习

- 低功耗设计
- 芯片设计流程

建议：

芯片 + AI 可以了解，但不建议一开始作为主线。更现实的入门切口是：

```text
FPGA 矩阵乘法加速
边缘 AI 模型部署
TinyML
嵌入式推理
```

## 方向选择建议

按“门槛适中、与 AI 结合自然、适合电子电气背景”排序：

```text
第一选择：电子 + 边缘 AI / 智能硬件
第二选择：计算机工程硬件分支 + 边缘 AI / 硬件加速
第三选择：电气 + AI 预测 / 故障诊断
第四选择：通信 + AI 信号处理
第五选择：芯片 + AI
```

更适合当前阶段的组合路线：

```text
电子硬件基础
+ 嵌入式 / 计算机组成
+ Python 后端
+ 数据库
+ 数据分析
+ 边缘 AI
+ LangGraph Agent
```

当前校园管理项目后续可以往这个方向扩展：

```text
校园管理系统
-> 校园数据分析平台
-> 加入教室环境传感器模拟数据
-> 做异常检测
-> 用 LangGraph 生成分析报告
```

这样项目就不只是管理系统，而是“电子 + AI”路线的训练起点。
